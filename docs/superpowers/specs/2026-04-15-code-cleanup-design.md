# DogBot 代码清理设计

日期：2026-04-15

## 1. 背景

为了先跑通 `QQ -> Agent` 和 `微信 -> Agent` 的完整链路，当前仓库里留下了一批偏临时性的实现：

- `wechatpadpro_adapter` 把 Webhook 接入、消息标准化、命令匹配、去重、执行、回复等逻辑堆在少量文件里，尤其是 `app.py`
- `scripts/` 目录里的脚本存在重复的环境变量加载、重复的 Compose 解析、重复的错误处理，以及平台逻辑和部署逻辑混杂的问题
- `agent-runner` 整体方向已经比较清楚，但入口与配置边界还不够干净，脚本层和服务层之间仍然有一些临时性耦合

这些问题不会立刻阻断功能，但会直接提高后续两项工作的难度：

1. 易用性整理
2. 去除 AstrBot 依赖

因此，在进入下一阶段之前，需要先做一次中等力度的结构清理。

## 2. 目标

本次清理只做结构整理，不追求引入新能力。

目标有四个：

1. 清理已经验证有效的代码路径，去掉明显的脏实现和临时补丁堆积
2. 让 `wechatpadpro_adapter`、`agent-runner`、`scripts/` 的职责边界更清楚
3. 为后续“去掉 AstrBot、统一平台 adapter、整理一键部署”打基础
4. 保持当前已经可用的微信链路、Claude 执行链路和现有部署链路不被破坏

## 3. 非目标

本次不做以下事情：

- 去除 AstrBot
- 新增 `qq-adapter`
- 调整用户侧配置模型
- 重写部署交互流程
- 引入历史消息持久化或长期记忆
- 处理平台兼容性问题

## 4. 清理范围

本次只处理三块：

- `wechatpadpro_adapter/`
- `agent-runner/`
- `scripts/`

`astrbot/` 相关代码本轮不处理，因为它后续会被替换掉。

## 5. 方案选择

### 方案 A：保守清理

只调整命名、注释、局部重复代码，不改变模块边界。

优点：

- 风险最小
- 修改量最小

缺点：

- 大部分结构问题还在
- 后续做易用性整理时还要再拆一次

### 方案 B：中等清理

重新整理模块边界，拆掉大文件里的混杂职责，但不直接切到最终架构。

优点：

- 可以明显降低后续重构成本
- 能保留当前有效能力
- 风险和收益比较平衡

缺点：

- 仍然会保留一部分当前架构痕迹

### 方案 C：激进清理

直接按最终目标架构重写三块代码。

优点：

- 最接近后续目标

缺点：

- 改动过大
- 容易把“清理”和“重构”混在一起
- 失败成本高

### 选择

本次采用方案 B：中等清理。

## 6. 模块设计

### 6.1 `wechatpadpro_adapter`

当前问题：

- `app.py` 既处理 FastAPI 路由，又处理事件过滤、命令规则、去重、执行和回复
- 文本规整化和命令匹配规则分散在多个函数里
- 事件处理流程虽然能跑，但不利于后续抽出通用 adapter 能力

本次整理后，`wechatpadpro_adapter` 收敛成五层：

1. 配置层
   - 负责读取环境变量并归一化配置
2. 事件标准化层
   - 负责把 WeChatPadPro 原始 payload 解包成统一可处理的事件对象
3. 触发策略层
   - 负责私聊/群聊命令规则、文本规整化、去重、忽略控制消息
4. 处理器层
   - 负责调用 `agent-runner`、拼装回复、发送微信消息
5. Web 框架入口层
   - 只保留 FastAPI 路由和依赖装配

整理后原则：

- `app.py` 只保留 HTTP 入口和依赖装配
- 消息逻辑不再直接堆在路由里
- 后续新增 `qq-adapter` 时，可以复用“命令策略”和“runner 调用”的设计思路

### 6.2 `scripts/`

当前问题：

- 每个脚本都自己做一遍 env 文件解析
- 每个脚本都自己判断 Compose 命令
- 错误输出风格不统一
- 生命周期脚本、平台配置脚本、调试脚本之间边界不清楚

本次整理后，`scripts/` 收敛成三类：

1. 生命周期脚本
   - 启动、停止
2. 平台配置脚本
   - NapCat / WeChatPadPro 的接入和登录准备
3. 调试工具脚本
   - 结构检查、主动发消息、网络策略、冒烟测试

同时引入统一的 shell 公共库，负责：

- env 文件定位与加载
- Compose 命令解析
- 基础错误输出
- 常用路径计算

整理后原则：

- 共享逻辑进入公共库
- 业务逻辑留在各自脚本
- 不再在每个脚本里复制一份启动样板

### 6.3 `agent-runner`

当前问题：

- 核心能力已经成形，但入口与配置边界还不够明确
- 服务启动和脚本环境变量之间有一些临时性耦合
- 配置层和 HTTP 入口层的职责还可以更清楚

本次整理后，`agent-runner` 保持当前功能边界不变，但收敛成更清楚的结构：

1. 配置层
   - 只负责读取并校验环境变量
2. 运行时层
   - 负责拼装主服务和内置代理服务
3. 核心能力层
   - Docker 执行
   - 会话持久化
   - 消息发送
   - HTTP 处理

整理后原则：

- 不改核心运行模型
- 不追求重写服务层
- 重点清理入口与配置边界

## 7. 执行顺序

本次按下面顺序执行：

1. `wechatpadpro_adapter`
2. `scripts/`
3. `agent-runner`

原因：

- `wechatpadpro_adapter` 当前最脏，且最能提前抽出未来 adapter 的边界
- `scripts/` 是后续“一键部署”重构的基础
- `agent-runner` 应该最后收敛，避免前两步刚改完又被入口配置变化打断

## 8. 验证要求

本次清理后至少要保证：

- 微信 adapter 仍然能接收有效文本消息并调用 `agent-runner`
- `agent-runner` 仍能执行 Claude，并通过内置代理访问上游
- 当前部署脚本和停止脚本仍可运行
- 现有测试和结构检查继续通过

验证重点不是兼容旧脏实现，而是保住当前已经有效的主链路。

## 9. 风险与约束

本次最主要的风险有三个：

1. 清理 `wechatpadpro_adapter` 时把当前已生效的微信命令规则弄坏
2. 清理 `scripts/` 时影响现有部署入口
3. 清理 `agent-runner` 入口时影响脚本对它的启动方式

对应的控制方式：

- 每一层整理后立即跑对应测试
- 尽量先拆边界，再调实现
- `scripts/` 的对外入口文件名和基本调用方式保持不变

## 10. 结果要求

本次清理完成后，代码应达到下面状态：

- `wechatpadpro_adapter` 不再依赖一个过大的 `app.py`
- `scripts/` 中共享逻辑集中管理
- `agent-runner` 的配置与入口边界更明确
- 仓库为下一步“去掉 AstrBot + 整理一键部署”做好准备
